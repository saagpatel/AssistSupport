# ADR 0011 — Tauri capabilities least-privilege audit

**Status:** Accepted
**Date:** 2026-04-22
**Supersedes:** —
**Superseded by:** —

## Context

The app grants Tauri capabilities (plugin + custom command) to the main window via
`src-tauri/capabilities/default.json`, backed by the custom permission zones
defined in `src-tauri/permissions/default.toml` (ADR 0003 established the zoning).
Before this audit, no end-to-end review had confirmed that the surface exposed
to the webview was the minimum required for the actual feature set.

## Audit methodology

Three sets were extracted and diffed to identify drift:

1. **Registered** — every command declared in `src-tauri/src/commands/registry.rs`
   via the `tauri::generate_handler![…]` macro (250 commands).
2. **Permitted** — every command name allowed under one of the custom permission
   identifiers in `src-tauri/permissions/default.toml` (250 commands).
3. **Invoked** — every snake_case string literal appearing in production frontend
   source (`src/**/*.{ts,tsx}` excluding `*.test.*`, `src/test/`, and
   `e2eTauriMock`) whose value matches a registered command name.

Set operations computed with `comm(1)` after sorting. The invoked set was also
cross-checked against Rust-side references (`src-tauri/src/**/*.rs`) to rule
out commands triggered from backend setup hooks rather than the frontend.

## Findings

### Permission model is already principled

- Zero wildcard grants. No `*-allow-all`, no `fs:allow-read-all`, no
  `shell:allow-execute`. Every command permission is an explicit allow-list.
- No `tauri-plugin-fs`, no `tauri-plugin-shell`, no `tauri-plugin-http`.
  The only plugins registered are `dialog` (used) and `opener` (unused — see
  below).
- The 10 custom zones (`startup-core`, `diagnostics-and-recovery`,
  `vector-and-model-runtime`, `knowledge-base`, `drafts-and-templates`,
  `customization-and-workspace`, `integrations-and-secrets`, `search-sidecar`,
  `jobs-and-evals`, `operations-and-analytics`) each enumerate their commands
  explicitly and match the grouping in `registry.rs`.
- Zero drift: every registered command appears in exactly one permission zone,
  and every permitted command is registered.

### Actionable findings (applied in this PR)

1. **`greet` command is Tauri template cruft.** Introduced by `create-tauri-app`
   scaffolding, never referenced from frontend or backend. Removed from
   `registry.rs`, `default.toml`, `app_core_commands.rs`, and the orphaned
   duplicate in `commands/mod.rs`. The autogenerated permission file at
   `src-tauri/permissions/autogenerated/greet.toml` is a build artifact and
   regenerates from the registry on the next build.
2. **`tauri-plugin-opener` is not used anywhere.** Registered in `lib.rs` via
   `.plugin(tauri_plugin_opener::init())` and granted `opener:default` in the
   capability, but the only `opener` occurrence in source code is the unrelated
   HTML `rel="noopener noreferrer"` anchor attribute — no call to `openUrl` or
   `revealItemInDir`. Removed the dep from `Cargo.toml`, the registration from
   `lib.rs`, and the permission from `capabilities/default.json`. This also
   shrinks the binary and attack surface.

### Findings not addressed in this PR (follow-up candidates)

- **70 commands are permitted but never invoked from production frontend.**
  Verified examples include `start_kb_watcher`, `stop_kb_watcher`,
  `is_kb_watcher_running`, `has_hf_token` / `set_hf_token` / `clear_hf_token`,
  `set_search_api_token` / `clear_search_api_token`,
  `preview_backup_import`, `process_ocr`, `greet`, `use_playbook`,
  `list_jobs` / `get_job` / `get_job_counts` / `get_job_logs`. Each should be
  reviewed to determine whether it is (a) planned-but-unwired, (b) dead from a
  deferred feature, or (c) still reachable via a call path my static analysis
  missed. Batch candidates for removal in a follow-up; do **not** delete blindly.
- **Plugin-level grants include sub-permissions we do not use.** `core:default`
  aggregates nine sub-permissions, of which `core:tray:default`, `core:menu:default`,
  `core:image:default`, `core:window:default`, and `core:webview:default`
  correspond to APIs the frontend does not call (no `@tauri-apps/api/menu`,
  `/api/image`, `/api/window`, or `/api/webview` imports, no tray or menu
  registration on the Rust side). Narrowing `core:default` → the explicit set
  `core:event:default`, `core:app:default`, `core:path:default`,
  `core:resources:default` is theoretically possible but carries a non-trivial
  regression risk (transitive Tauri internals may depend on the others).
  Defer until someone can trial on a branch with end-to-end UI verification.
- **The 10 custom zones bundle commands that differ in sensitivity.** For
  example `integrations-and-secrets` grants token getters/setters/clearers
  alongside unauthenticated `is_jira_configured` / `list_integrations` reads.
  Splitting could enable webview segmentation (`isolation` pattern) in the
  future, but there is only one window today so the benefit is theoretical.

## Decision

1. Remove the `greet` template command and associated permission entry.
2. Remove the unused `tauri-plugin-opener` dependency, plugin registration,
   and capability grant.
3. Record the audit methodology (above) so it can be re-run as the command
   surface evolves.
4. Defer the 70-dead-commands sweep and the `core:default` narrowing to
   scoped follow-up PRs, both of which need per-item verification to avoid
   the false-positive pattern flagged in prior audits.

## Consequences

- **Positive:** Smaller attack surface (one fewer plugin, one fewer template
  command). Smaller binary. Capability file now lists only actually-used
  permissions. Future auditors have a documented methodology and a known
  follow-up list.
- **Neutral:** No behavioral change for end users — the removed plugin and
  command were unreachable from the UI.
- **Risk:** Very low. `cargo check --all-targets` passes with no warnings.
  The only way to break runtime is if someone adds frontend code invoking
  `openUrl` or `greet` — they would fail with a clear "command not found"
  or "permission denied" error, surfaced immediately in dev.

## Methodology artifacts

For reproducing the audit on a future commit:

```bash
# Registered commands
grep -oE '::[a-z_][a-z_0-9]+[,[:space:]]*$' src-tauri/src/commands/registry.rs \
  | sed -E 's/^:://; s/[,[:space:]]+$//' | sort -u > /tmp/registered.txt

# Permitted commands
grep -oE '"[a-z_][a-z_0-9]+",' src-tauri/permissions/default.toml \
  | sed 's/^"//; s/",$//' | sort -u > /tmp/permitted.txt

# Likely-invoked commands (conservative — production frontend only)
find src -name '*.ts' -o -name '*.tsx' \
  | grep -vE '\.test\.|/test/|e2eTauriMock' \
  | xargs grep -hoE '["'"'"'\`][a-z_][a-z_0-9]+["'"'"'\`]' \
  | sed -E 's/^["'"'"'\`]//; s/["'"'"'\`]$//' \
  | sort -u > /tmp/string-literals.txt
comm -12 /tmp/registered.txt /tmp/string-literals.txt > /tmp/invoked.txt

# Drift
comm -23 /tmp/registered.txt /tmp/permitted.txt   # registered, not permitted
comm -13 /tmp/registered.txt /tmp/permitted.txt   # permitted, not registered
comm -23 /tmp/permitted.txt /tmp/invoked.txt      # permitted, not invoked
```

Caveat: the "invoked" set is conservative — a command that the frontend never
calls may still be legitimate (e.g., invoked from a menu item, a Tauri event
callback, or planned for imminent use). Manual confirmation is required before
deleting any command from that list.
